## Story-by-Story Analysis

### Story 1.1 — Project Foundation & Infrastructure

**Key Review Patches:**
- `resetActiveSession` must use functional `set(state => ({...}))` form — direct set used initially
- Network errors propagated as `TypeError` instead of `ApiError` — `fetch` needed try/catch wrapper
- 204/empty responses caused JSON parse failure — content-type check required before `res.json()`
- 401 redirect looped infinitely when already on `/login` — pathname guard added
- `BASE_URL` unchecked when `VITE_API_BASE_URL` undefined — startup assertion added

**Key Deferred Items:**
- `QueryClient` has no `retry`/`staleTime` config — may cause redundant 4xx retries
- Store allows incoherent auth state (e.g., `status='authenticated'` with `neurologist=null`)
- `LoadingSpinner` has no visual animation — build passes but visual polish missing

**Notable Event:** `NeurologistDashboard.tsx` had 2 pre-existing TypeScript errors (`TS6133`) that blocked `npm run build`. Dev Notes said "do not touch" — escalated to Mauri for approval. Two minimal patches applied (`_unread` rename, `export` added) with no behavior change. Build passed. **Escalation process worked correctly.**

**Breakthrough:** TypeScript constraint `erasableSyntaxOnly: true` discovered and documented — prevented recurrence in Stories 1.2 and 1.3.

---

### Story 1.2 — Application Routing & Layout Shell

**Key Review Patches:**
- Concurrent `requireAuth` invocations raced on shared Zustand state — dedupe guard via `authCheckInFlight` promise
- Initials computation crashed on names with consecutive/leading/trailing spaces — edge case fix

**Key Deferred Items:**
- `/login` route not guarded for authenticated users → resolved in Story 1.3
- `setAuth(Partial<AuthSlice>)` allows incoherent state — pre-existing design decision

**Notable Observation:** `ActiveSessionBanner` used `className="active-session-banner"` instead of the inline-style blueprint variant — minor divergence, no impact, Story 3.1 replaces the component entirely.

**Debug Log:** Clean run — no build errors on first attempt. Learnings from Story 1.1 applied immediately.

---

### Story 1.3 — Login & Logout

**Key Review Patches:**
- `redirectIfAuthenticated` enhanced to async loader calling `getMe()` on `'loading'` state — cold-load authenticated users now correctly redirected to `/patients`
- `VITE_API_BASE_URL` undefined silently cast to `string` — explicit `string | undefined` check added, buttons disabled when `apiBase` falsy
- `queryClient.cancelQueries()` added before `queryClient.clear()` — prevents observer re-fetches during logout
- AC6 login guard strengthened (above)

**Deferred Items Resolved from Story 1.2:**
- `/login` route guard for authenticated users ✅
- `LogoutButton` in AppShell ✅

**Notable:** Blueprint included `<LogoutButton style={{ marginLeft: 'auto' }} />` but the component doesn't accept a `style` prop — wrapped in `<div style={{ marginLeft: 'auto' }}>` instead. Minor blueprint inconsistency caught in implementation.

## Patterns Across All Stories

### Recurring Strengths

1. **Service layer discipline maintained**: Zero `fetch` calls outside `api.ts` across all 3 stories — NFR-6 compliance 100%.
2. **Code review catches real bugs**: 10 patches across 3 stories — redirect loops, JSON parse failures, race conditions, undefined env vars. All caught pre-production.
3. **Escalation process functional**: Ambiguous decisions (NeurologistDashboard deviation) escalated to Project Lead before proceeding.
4. **Learning propagation**: `erasableSyntaxOnly: true` discovered in Story 1.1 → documented → not hit in Stories 1.2 or 1.3.
5. **Deferred items tracked**: `deferred-work.md` maintained throughout. 2 deferred items from 1.2 resolved in 1.3.

### Recurring Challenges

1. **Project baseline constraints not documented in planning artifacts**: TypeScript constraints, pre-existing TS errors in `NeurologistDashboard.tsx`, and CSS class availability required runtime discovery rather than spec-time knowledge.
2. **Stub-then-rewrite pattern**: `LoginPage.tsx` created as stub in Story 1.2 and completely rewritten in Story 1.3. Added overhead without clear architectural benefit.
3. **Blueprint/implementation divergence**: Small but recurring gaps between Dev Notes blueprints and actual implementations (inline style vs className, LogoutButton style prop). Not harmful but worth tightening in future epics.
4. **No test framework**: Only verification gate is `npm run build`. Subtle behavioral bugs (race conditions, state management edge cases) require code review to catch rather than automated tests.
