## Patterns Across All Stories

### Recurring Strengths

1. **Blueprint-following = clean builds**: Stories that followed blueprints verbatim (3.1, 3.3, 3.4) had the fewest patches. The investment in detailed Dev Notes pays off measurably.

2. **Service layer discipline — 100%**: Zero `fetch` calls outside `api.ts` across all 4 stories. NFR-6 maintained throughout the most complex epic yet.

3. **Learning propagation working**: Epic 2 patterns (TanStack v5, Zustand selector form, no snake_case in components) had zero re-discovery in Epic 3.

4. **Patch trend improving**: 4 → 9 → 1 → 2. After the 3.2 peak, the team applied learnings and 3.3 and 3.4 emerged nearly clean.

5. **`destroyed` flag pattern**: The memory leak prevention pattern in `useSessionWebSocket` is a reference implementation for any future hook with async effects.

### Recurring Challenges

1. **Cross-story side effects escape per-story review**: The `SessionCompletionToast` double-mount required seeing both Story 3.2 (added to AppShell) and Story 3.3 (still in SessionMonitorPage) simultaneously. Per-story code review cannot catch this class of bug. Need explicit cross-component verification step.

2. **Long-outstanding deferred items carry forward silently**: A10 (fix 401 in `api.get`) was marked high-priority in Epic 2 retro and not addressed in any of the 4 Epic 3 stories. Without a forcing function in Dev Notes, these items slip.

3. **Deferred items accumulating without triage**: `deferred-work.md` now holds 35 items across Epics 1–3. Risk level varies widely with no active prioritization.

## What Was Challenging

- A10 (401 fix) carried forward again — 2 epics without resolution

## Key Takeaways

1. **Cross-story component moves require explicit verification**: When a code review patches a component from file A to file B, the next story must verify A no longer imports it. This is a checklist item for Dev Notes "Current State of Files" section.

2. **Prerequisite fixes work when they're the explicit first task**: A1 and A5-A7 were delivered because they were the *first* task of 3.2 with explicit formatting. A10 was not — and it slipped again.

3. **Blueprint-following velocity is real**: 3.4 had 2 patches and clean implementation despite being a third-party integration. Preparation eliminates uncertainty.

4. **`MLField<T>` vs `string | null` distinction is architecture-critical**: For live session metrics, `MLField<T>` pending/resolved/error is correct. For historical dashboard data, `string | null` with `'—'` fallback is correct. Story 4.3 must be explicit about this distinction.
