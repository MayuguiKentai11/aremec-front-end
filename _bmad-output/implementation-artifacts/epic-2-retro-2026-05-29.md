## Story-by-Story Analysis

### Story 2.1 — Patient Registration Form

**Key Review Patches:**
- Age minimum set to 50 (clinical constraint for EA/MCI patient population)
- RAVLT and SART bounds validation added: RAVLT 0–75, SART 0–100
- `Number.isFinite()` guard added — prevents `Infinity` and `NaN` from passing Zod's `.number()` check

**Key Deferred Items:**
- `defaultValue=""` on selects without `useForm({ defaultValues })` — works now, becomes defect if `reset()` ever added
- `toCamel(undefined)` on non-JSON API response — TypeError propagates to `mutation.error`; message text may be confusing
- No `AbortController` — orphaned mutations complete async on navigation; double `navigate` is no-op in React Router
- `navigate` fires before `invalidateQueries` settles — stale data flash on patient list observable in production

**Notable Event:** Project ships Zod v4.4.3, not v3 as specified in blueprints. `z.coerce.number()` with React Hook Form required splitting schema into Input/Output types. Resolved in-story without escalation.

**Build Quality:** First build passed after Zod adaptation. No TypeScript errors at close.

### Story 2.2 — Patient List & Search

**Key Review Patches:**

- Status filter `<select>` missing `aria-label` — `aria-label="Filtrar por estado"` added

### Story 2.3 — Patient Profile Page

**Key Review Patches:**
- `usePatient` missing `enabled: !!id` — query fired with empty string when `id` was undefined, sent `GET /patients/` to backend
- Orphaned "Sesión activa" tab — when session ends, `showSessionTab` becomes false but active tab state stayed on `'sesion-activa'`; `useEffect` added to reset tab on `showSessionTab` → false
- Tab state not reset on patient navigation — navigating directly between `/patients/A` and `/patients/B` kept the previous tab active; `useEffect` on `[id]` added to reset to `'resumen'`
- `getPatient` built URL without `encodeURIComponent` — patient IDs with special characters could corrupt the path segment
- Background refetch error destroyed stale data — `if (error || !patient)` replaced with separate initial-load guard (`if (!patient)`) and inline error banner when stale data is still available

**Key Deferred Items:**
- `status ?? 'active'` fallback is dead code per TypeScript types; empty string from API silently mis-classifies patient status
- `401` in `api.get` resolves `undefined as never` — TanStack Query treats as success; `patient` becomes undefined, shows `<ErrorMessage error={null}>` during auth redirect
- `age` field could be a float from API — renders as "75.5 años" with no rounding
- `useAppStore(s => s.activeSession.sessionId)` lacks optional chaining — resolved by Epic 3 Story 3.1 store expansion

**Notable Discovery:** `activeSession.sessionId` is compared to `patient.id` in the tab visibility check (`showSessionTab = activeSessionId === id`). Story 3.1's AC specifies that `POST /sessions` returns a session UUID which is stored in `activeSession.sessionId`. These two UUIDs are different identifiers — the check would always be false. **The store needs a `patientId` field added in Story 3.1 before implementing session open.** See Significant Discoveries.

**Build Quality:** Build passed in 267ms (first compile). 0 TypeScript errors at close.

### Recurring Challenges

1. **Accessibility not in the planning radar**: WCAG violations (keyboard navigation, aria-labels) were found in code review, not during implementation. The dev agent does not consider accessibility unless explicitly prompted in Dev Notes or Acceptance Criteria. In a clinical portal serving neurologists who may have varying accessibility needs, this is not optional.

3. **QueryClient retry configuration missed — second epic running**: Committed as Action Item 3 in Epic 1 retro (owner: Amelia). Not addressed in any of the 3 Epic 2 stories. The risk is real: without `retry: false` on mutations, failed `createPatient` calls may be retried, potentially creating duplicate patients in the backend.
4. **Deferred items accumulating without triage**: `deferred-work.md` now holds 22 items. Some overlap across stories. Several items that were low-priority in Epic 2 become medium or high priority in Epic 3 (see Epic 3 Preparation).

## Significant Discoveries

### Store Shape Conflict — `activeSession.sessionId` vs `patientId`

**The conflict:**
- Story 2.3 spec: `store.activeSession.sessionId === patient.id` — the tab visibility check compares `sessionId` to the patient's UUID
- Story 3.1 AC: `POST /sessions` returns a session UUID, stored in `activeSession.sessionId`
- These are two different UUIDs — the comparison would always be false, and the "Sesión activa" tab would never appear

**Root cause:** The spec uses `sessionId` to mean two different things: the patient's UUID (in Story 2.3 tab visibility) and the session entity's UUID (in Story 3.1 lifecycle management).

**Resolution required in Story 3.1:**

```typescript
// Current store — missing patientId
type ActiveSessionSlice = {
  sessionId: string | null    // session UUID — for API and WebSocket
  startedAt: Date | null
  wsStatus: 'connected' | 'reconnecting' | 'polling' | 'disconnected'
}

// Required store — add patientId
type ActiveSessionSlice = {
  sessionId: string | null    // session UUID — POST /sessions response
  patientId: string | null    // patient UUID — for tab visibility check
  startedAt: Date | null
  wsStatus: 'connected' | 'reconnecting' | 'polling' | 'disconnected'
}
```

On session open (Story 3.1): store both `sessionId = response.session_id` and `patientId = patient.id`.
On session close: reset both to `null`.
In `PatientProfilePage`: `const showSessionTab = useAppStore(s => s.activeSession.patientId) === id`.

This change is entirely contained within Story 3.1 — `PatientProfilePage` is already wired correctly and just needs the selector updated.
